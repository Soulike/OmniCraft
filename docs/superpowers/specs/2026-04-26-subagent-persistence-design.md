# Subagent Persistence

## Problem

Top-level chat and coding agents already persist their full history to disk through the shared `Agent` persistence flow. Dispatched subagents currently do not, because `GeneralSubAgent` and `ExploreSubAgent` are constructed without a `sessionsDir`. Their `AgentSseLog` stays in memory only, and their `AgentSnapshot` is never written.

The parent session does save forwarded subagent events in its own `sse-events.jsonl`, so the UI can replay the subagent transcript as part of the parent conversation. That is not enough for future resumption, because the subagent's own `llmSession.messages` are not saved anywhere.

## Goals

- Persist every dispatched regular subagent using the same `Agent` persistence path as top-level agents.
- Save subagent `snapshot.json`, `metadata.json`, and `sse-events.jsonl` under the parent session directory.
- Keep subagent snapshots identical to normal `AgentSnapshot` objects. The history source is still `snapshot.llmSession.messages`.
- Avoid custom subagent metadata in this phase.
- Avoid implementing subagent resume behavior in this phase.
- Keep existing subagent title behavior unchanged.

## Non-Goals

- Do not add a `SubagentStore` yet. Persistence does not require memory caching or typed restore.
- Do not add subagent type metadata.
- Do not add `resume_subagent`, `read_subagent_history`, or `list_subagents` tools.
- Do not add top-level API routes or frontend UI for subagent sessions.
- Do not change the parent session history model.
- Do not alter `CodingSubAgent` unless it is wired into `dispatch_agent` in a separate change.

## Design Decisions

| Decision              | Choice                                                         | Rationale                                                                                                |
| --------------------- | -------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------- |
| Persistence mechanism | Reuse `AgentOptions.sessionsDir`                               | The base `Agent` already persists snapshots and SSE logs when `sessionsDir` exists.                      |
| Storage location      | `<parent sessionsDir>/<parent id>/subagents/<subagent id>/`    | Keeps child agents scoped to their parent and deleted with the parent session.                           |
| Metadata              | Reuse existing `metadata.json` generated by `AgentPersistence` | Enough for persistence and debugging. Typed resume metadata is deferred.                                 |
| Store abstraction     | No `SubagentStore` in this phase                               | There is no need for caching or disk restore until resume exists.                                        |
| Title behavior        | Keep current `Agent` title generation                          | Subagents can generate titles the same way as other agents.                                              |
| Resume readiness      | Save normal snapshots only                                     | Future resume can load a snapshot and construct the requested subagent type supplied by the resume tool. |

## Storage Layout

For a persisted coding parent session:

```text
$DATA_DIR/coding-sessions/<parent-agent-id>/
  snapshot.json
  metadata.json
  sse-events.jsonl
  subagents/
    <subagent-id>/
      snapshot.json
      metadata.json
      sse-events.jsonl
```

For a persisted chat parent session, the same layout is used under `$DATA_DIR/sessions/<parent-agent-id>/subagents/`.

No custom `subagent.json` is introduced in this phase.

## Backend Design

### Agent Context

`dispatch_agent` needs enough parent context to compute the child sessions directory. Add these fields to `ToolExecutionContext`:

```typescript
readonly agentId: string;
readonly sessionsDir: string | null;
```

Populate them in `Agent.executeTool()` from the current agent instance:

```typescript
const context: ToolExecutionContext = {
  callId: toolCall.callId,
  agentId: this.id,
  sessionsDir: this.sessionsDir,
  // existing fields unchanged
};
```

No agent type is needed for persistence-only support.

### Subagent Sessions Directory

In `dispatch_agent`, compute the subagent sessions directory from parent context:

```typescript
const subagentSessionsDir = context.sessionsDir
  ? path.join(context.sessionsDir, context.agentId, 'subagents')
  : undefined;
```

If `context.sessionsDir` is `null`, keep the current in-memory behavior. This preserves tests and any future non-persisted agent use.

### Subagent Constructors

Update regular subagents to accept an optional `sessionsDir`:

```typescript
export class GeneralSubAgent extends Agent {
  constructor(
    getConfig: () => Promise<LlmConfig>,
    workingDirectory: string,
    sessionsDir?: string,
  ) {
    super(getConfig, {
      toolRegistries: [
        CoreToolRegistry.getInstance(),
        FileToolRegistry.getInstance(),
        WebToolRegistry.getInstance(),
        BashToolRegistry.getInstance(),
      ],
      skillRegistries: [CoreSkillRegistry.getInstance()],
      baseSystemPrompt:
        'You are a helpful assistant working on a delegated subtask. ' +
        'After completing your task, provide a concise summary of what you did and the results.',
      getMaxToolRounds: async () => {
        const settings = await settingsService.getAll();
        return settings.agent.maxToolRounds;
      },
      workingDirectory,
      sessionsDir,
    });
  }
}
```

`ExploreSubAgent` receives the same constructor shape and passes `sessionsDir` to `super()`.

No abstract `SubAgent` class is required for this phase. It can be added later when resume introduces shared typed restore behavior.

### Subagent Factory

Update `createSubAgent()` to accept and forward `sessionsDir`:

```typescript
export function createSubAgent(
  agentType: SubAgentType,
  getConfig: () => Promise<LlmConfig>,
  workingDirectory: string,
  sessionsDir?: string,
): Agent {
  switch (agentType) {
    case SUB_AGENT_TYPE.GENERAL:
      return new GeneralSubAgent(getConfig, workingDirectory, sessionsDir);
    case SUB_AGENT_TYPE.EXPLORE:
      return new ExploreSubAgent(getConfig, workingDirectory, sessionsDir);
  }
}
```

`dispatch_agent` passes `subagentSessionsDir` when constructing the subagent.

### Persistence Lifecycle

Once a subagent receives `sessionsDir`, existing `Agent` behavior handles persistence:

- Constructor writes the initial `snapshot.json` and `metadata.json` synchronously.
- Each SSE event appends to `sse-events.jsonl`.
- The `done` event triggers a snapshot write with completed `llmSession.messages`.
- Session title generation stays unchanged and updates the snapshot and metadata when it finishes.

This means the parent session continues to store forwarded subagent events, and the subagent now also stores its own internal history.

## Future Resume Path

This design intentionally saves enough data for a future resume tool without implementing that tool now.

Future resume can work by requiring the caller to provide the subagent type:

```typescript
const snapshot = await agentPersistence.loadSnapshot(subagentSessionsDir, id);

switch (subagentType) {
  case SUB_AGENT_TYPE.GENERAL:
    return new GeneralSubAgent(
      getConfig,
      snapshot.options.workingDirectory ?? parentWorkingDirectory,
      subagentSessionsDir,
      snapshot,
    );
  case SUB_AGENT_TYPE.EXPLORE:
    return new ExploreSubAgent(
      getConfig,
      snapshot.options.workingDirectory ?? parentWorkingDirectory,
      subagentSessionsDir,
      snapshot,
    );
}
```

The exact constructor signature for snapshot restore is deferred until resume is implemented. The persistence-only change should not add snapshot parameters unless tests need an immediate restore helper.

## Risks

- Subagent title generation adds the same light-model title call that top-level agents use. This is accepted for consistency.
- If the server crashes before the subagent `done` event, `sse-events.jsonl` may contain more streamed UI events than `snapshot.json` knows about. This is the same alignment model used for top-level agents and is reconciled on restore.
- Without subagent type metadata, future resume must be called with the correct subagent type. This is acceptable because resume is out of scope for this phase.

## Success Criteria

- Dispatching a regular subagent from a persisted parent session creates a child directory under `subagents/`.
- The child directory contains `snapshot.json`, `metadata.json`, and `sse-events.jsonl` after the subagent completes.
- The child `snapshot.json` contains the subagent's own `llmSession.messages`.
- Existing parent session replay behavior remains unchanged.
- No new subagent metadata or resume API is introduced.
