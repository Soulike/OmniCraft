# Subagent Nicknames Design

**Date:** 2026-06-14
**Issue:** [#262 â€” Consider to Use Readable Names for Subagent Resumption Instead of UUID](https://github.com/Soulike/OmniCraft/issues/262)

## Problem

Subagents are identified by UUIDs everywhere, including on the surface the LLM
uses to resume them. The LLM dispatches a subagent, receives a UUID, and must
echo that UUID back to `resume_agent` to send follow-up work. High-entropy,
semantically empty UUIDs are easy for a model to transpose or hallucinate across
turns. Each wrong id is a failed `resume` round-trip (`resume-agent-tool.ts`
returns an error for an unknown id).

Replacing the LLM-facing handle with a readable nickname (e.g. `crimson-otter`)
gives the model a semantic anchor that is far harder to corrupt, and incidentally
makes the frontend disclosure and human debugging more legible.

## Key Constraint: Resume Is Registry-Only

`resume_agent` resolves exclusively against the in-memory `SubagentRegistry`
(`resume-agent-tool.ts` â†’ `subagentRegistry.get()`). The registry is a bounded
(`DEFAULT_MAX_LIVE_SUBAGENTS = 10`) LRU `Map`; once an entry is evicted
(`subagent-registry.ts` `evictIfNeeded`), the subagent can no longer be resumed
and the tool returns "dispatch a new one". There is **no path that rehydrates a
historical subagent from disk for resumption.**

Therefore a nickname only carries meaning for the lifetime of a registry entry.
It does **not** need to be persisted: its useful lifetime exactly equals the
resumable lifetime. Snapshot and persistence layers are untouched.

## Concept & Ownership

A **nickname** is a human/LLM-readable handle for a live subagent.

- **In-memory only.** Stored on the `SubagentRegistry` entry, assigned at
  registration, discarded at eviction. No field added to `AgentSnapshot` or any
  persisted file.
- **The LLM-facing identity.** `dispatch_agent` returns it, `list_resumable_agents`
  prints it, `resume_agent` accepts it. The UUID stays fully internal: registry
  `Map` key, SSE `agentId` routing, and on-disk session directory names. The UUID
  is never shown to the LLM.
- **Unique within the live set.** Generated from a hand-rolled wordlist and
  regenerated on collision against the currently-registered nicknames. With at
  most 10 live subagents against a ~2500-combination space, collisions are rare.

The UUID remains the canonical internal key. The nickname is a parallel,
ephemeral, LLM-facing alias. UUID generation and identity (`agent.ts`
`crypto.randomUUID()`) are unchanged.

## Backend Changes

### New module: `agent-core/agent/state/nickname.ts`

- `adjectives: string[]` and `nouns: string[]` â€” hand-maintained, ~50 each,
  lowercase, profanity-free, hyphen-safe.
- `generateNickname(taken: ReadonlySet<string>): string` â€” produces
  `${adjective}-${noun}`, retrying on collision with `taken`. If the combination
  space is ever saturated (should not happen at 10 live max), fall back to
  appending a short numeric suffix so the function always terminates with a
  unique value.

### `SubagentRegistry` (`subagent-registry.ts`)

- `LiveSubagentRegistryEntry` and `LiveSubagentRecord` gain a `nickname: string`.
- `register()` signature accepts the nickname (generated by the caller at
  dispatch time â€” see decoupling note below) and stores it on the entry.
- New `getByNickname(nickname: string): LiveSubagentHandle | undefined` â€” resolves
  a live entry by its nickname. The existing UUID-keyed `Map` is retained; the
  registry exposes nickname resolution by scanning entries (live set is â‰¤10) or a
  secondary nicknameâ†’id index, whichever is cleaner in implementation.
- `list()` records expose `nickname`.

### Dispatch / register decoupling

Today registration happens in `onTurnStarted` (after the turn starts) so the
registry does not briefly treat the new subagent as idle. But the dispatch
**result** must contain the nickname. Resolution (approved):

1. Generate the nickname at dispatch time, **before** the turn starts, against
   the registry's current live nicknames.
2. Thread it into the start event and the tool result.
3. Still call `register()` in `onTurnStarted` at the existing point, passing the
   pre-generated nickname.

Nickname generation and registry insertion are thus slightly decoupled: generate
early, insert at the existing point. The brief window between generation and
insertion is single-threaded within one dispatch call, so no collision race
exists.

### `dispatch-agent-tool.ts`

- Generate the nickname (step 1 above) and pass it to `runSubagentTurn` /
  `registerSubAgent`.
- The success result content switches from `<subagent_id>${uuid}</subagent_id>`
  to the nickname (`subagent-turn-runner.ts` builds this content; see below).
- Tool description: follow-up work references the returned **name**, not an id.

### `resume-agent-tool.ts`

- Parameter `agentId: z.string() (uuid)` â†’ `name: z.string().min(1)` describing
  the subagent name to resume.
- Resolve via `context.subagentRegistry.getByNickname(name)`.
- Unknown/missing name â†’ same failure shape as today ("not available to resume,
  dispatch a new subagent if needed").
- Drop the UUID-format validation branch; there is no UUID on this surface anymore.

### `list-resumable-agents-tool.ts`

- `formatListResumableAgentsContent` prints `name: ${nickname}` instead of
  `id: ${id}`.
- `ListedResumableAgent` carries `nickname` (sourced from the registry record).

### `subagent-turn-runner.ts`

- `SubagentTurnResult` and the start-event plumbing carry the nickname so the
  dispatch/resume result content and the SSE start event both expose it.
- Success content uses the nickname as the resume handle in place of the
  `<subagent_id>` wrapper.

## SSE + Frontend Changes

### SSE schema (`packages/sse-events/src/schema.ts`)

- `sseSubagentStartPayloadSchema` gains `nickname: z.string().optional()`.
  Optional **solely** for backward compatibility with already-persisted
  `subagent-dispatch` / `subagent-resume` events written before this change. New
  events always include it.
- `subagent-output` and `subagent-complete` do not need the nickname (routing is
  by `agentId`).

### `useStreamChat.ts`

- In the `subagent-dispatch` / `subagent-resume` case, pass `nickname` through to
  the `subagent-dispatched` bus event.

### Frontend types (`StreamingMessageDisplay/types.ts`)

- `SubagentContent` and the `subagent-dispatched` event map entry gain
  `nickname?: string`.

### `useMessages.ts`

- Thread `nickname` from the bus event into the constructed `SubagentContent`.

### `SubagentDisclosureView.tsx`

- The id row becomes a **name** row.
- `AGENT_ID_LABELS` â†’ labels `Subagent` (dispatch) / `Resumed subagent` (resume).
- Value renders `nickname ?? agentId`: new sessions show the nickname; old
  replayed sessions with no nickname fall back to the UUID.

## Testing

Mirror existing `.test.ts` neighbors:

- **Nickname generator:** format (`adjective-noun`), uniqueness against a `taken`
  set, deterministic termination under saturation.
- **Registry:** `getByNickname` resolution, nickname stored on entry, collision
  regeneration against the live set.
- **Tools:** `dispatch-agent-tool`, `resume-agent-tool`, `list-resumable-agents-tool`
  updated for the name surface (dispatch returns a name, resume resolves a name,
  list prints a name, unknown name fails gracefully).
- **Frontend:** `SubagentDisclosureView.test.tsx` covers the name row plus the
  UUID fallback when `nickname` is absent.

## Out of Scope

- Persisting nicknames (snapshot/metadata/disk unchanged).
- Rehydrating evicted subagents for resumption.
- Changing UUID generation or any internal UUID usage (registry key, SSE routing,
  session directories).
- Showing both nickname and UUID to the LLM or in the frontend; the frontend
  shows the nickname only (UUID remains available in the SSE stream for debugging).
